RAG / SFT 資料製作指引

呢個分頁唔會生成、審核、儲存或發布資料。管理員要由已核對來源人手建立 JSONL，再按 Workstation 發布流程驗證、簽署同安裝。

先分清 RAG 與 SFT

  • RAG 儲存需要檢索、引用、更新或撤回嘅知識。 事實、規則、案例同引用放入 RAG，唔好靠模型參數記住。
  • SFT 教模型點樣回答，例如語氣、結構、分析步驟同拒答方式； 唔應用嚟背誦會改變或需要引用嘅知識。

開始前先喺「LLM 文字審核」確認來源已接受、匿名化、具使用權，而且內容仍然有效。 保留來源擁有人、版本／日期、授權依據、內容 SHA-256 同撤回聯絡方法。

RAG 資料製作流程

  1. 由規則、教練筆記、已授權案例或可靠參考資料建立 canonical 原文；唔好將 AI 生成摘要當成事實來源。
  2. 按語意切段：每段只處理一個主題，並要離開原文件後仍可獨立理解。 唔好將完整長文直接當成一段，亦唔好為增加數量而重複近似段落。
  3. 建立 documents.jsonl；一行一個 JSON object，而且只可包含 "id"、"title"、"text"、 "source_url"。
  4. id 必須穩定而且唯一；title 要交代主題及版本； text 要包含可直接引用嘅內容、適用範圍同必要限制。 公開來源使用 HTTPS，純內部來源可將 source_url 留空。 
{"id":"rules-judging-2026-001","title":"評判準則｜論證完整性｜2026","text":"來源：校隊規則第六節。評判應檢視主張、理由、機制及影響是否連貫。適用於正式比賽；資料不足時不可自行補作事實。","source_url":""}

完成後將 documents.jsonl 放喺 RAG bundle 根目錄，建立新版本同 manifest，核對檔案 hash，再按 Workstation runbook 離線簽署。舊版本唔可以原位改寫。

SFT 資料製作流程

  1. 每一行係一個完整訓練例子，以 "messages" 保存 "system"、"user"、"assistant" 訊息；只訓練 assistant 回覆。
  2. user 訊息要係真實、完整而可回答嘅任務；assistant 目標答案要由熟悉辯論訓練嘅人 親自審核，示範期望語氣、推理步驟、引用方式同資料不足時嘅處理。
  3. 原始講稿或逐字稿唔係天然問答配對。要先人手界定問題、所需上下文同理想答案， 唔可以將相鄰段落自動當成 user／assistant。
  4. 同一來源、同一場比賽或同一份稿只可落入一個 split，避免內容同時出現喺 train、validation 同獨立 eval。 
{"messages":[{"role":"system","content":"你係香港中學辯論教練；資料不足時要講明。"},{"role":"user","content":"請分析以下主張點解欠缺機制：……"},{"role":"assistant","content":"呢個主張講咗結果，但未交代由原因去到結果中間發生咩事。你可以先補充……"}]}

發布前檢查

  • 逐項核對授權、匿名化、來源、日期、適用範圍、事實準確度同撤回狀態。
  • 移除 duplicate、相互矛盾而無版本標示嘅內容、私人資料同無法追溯來源嘅主張。
  • 驗證每行 JSON 都可獨立 parse，ID 無重複，檔案 hash、版本同 manifest 一致。
  • 預留完全未參與製作嘅 eval cases，測試檢索命中、引用準確、拒答同私隱。
  • 只有 blind eval 通過，先簽署及安裝新 bundle；失敗就保留目前 active 版本。
